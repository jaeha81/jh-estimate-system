## 1. 인테리어 견적의 업무 흐름

```
도면/내역서 수령
    ↓
공종 분류 (대공종 → 중공종 → 소공종)
    ↓
물량 산출 (면적, 수량, 길이)
    ↓
단가 적용 (자재비 + 노무비 + 경비)
    ↓
내역서 작성 (엑셀 템플릿)
    ↓
검토/수정 (review_flag 항목 확인)
    ↓
입찰서 제출 (브랜드별 양식 변환)
```

## 3. 단가 구조

### 단가 구성 원칙

```
단가 = 자재비 + 노무비 + 경비

자재비: 재료 원가 × (1 + 손율%)
노무비: 인력 × 노임단가 × 생산성
경비: (자재비 + 노무비) × 경비율%
```

### 지역별 노임단가 편차 (2025 기준 개략)

| 지역 | 지수 | 비고 |
|------|------|------|
| 서울/수도권 | 100% | 기준 |
| 광역시 | 90~95% | 부산, 대구 등 |
| 지방 | 80~90% | 인력 수급 차이 |

### 공종별 손율 (자재 손실률)

| 공종 | 손율 | 이유 |
|------|------|------|
| 타일 | 5~15% | 절단 손실, 패턴 맞춤 |
| 도배지 | 10~20% | 창호/코너 손실 |
| 강마루 | 5~10% | 절단 + 불량 |
| 도료 | 5% | 용기 잔류 |

| 컬럼 | 영문키 | 설명 |
|------|--------|------|
| 순번 | seq | 항목 번호 |
| 공종 | category | 대/중/소공종 |

| 규격 | spec | 재료 규격, 두께 등 |

| 비고 | note | 특기사항 |

### 처리 규칙

- `review_flag = true`: confidence < 0.7이거나 항목명 불명확
- `item_name_raw`: LLM이 임의 추정 금지, 원본 텍스트 유지
- 수식 셀: openpyxl 감지 후 skip (덮어쓰기 금지)

## 7. AI 에이전트 설계 원칙

### confidence 임계값

- `>= 0.9`: 자동 확정 가능
- `0.7 ~ 0.89`: 주의 표시, 자동 처리 허용
- `< 0.7`: `review_flag = true`, 사용자 컨펌 필수

### 7섹션 프롬프트 구조

① 역할 정의 → ② 입력 형식(JSON) → ③ 출력 형식(JSON) → ④ 처리 규칙 → ⑤ 금지사항 → ⑥ 참고 데이터(keyword_dict 주입) → ⑦ 예외 처리

### 핵심 금지 원칙

- review_flag=true 항목 자동 확정 금지
- 불명확 항목 LLM 임의 추정 후 확정 금지
- 원본 엑셀 템플릿 덮어쓰기 금지 (복제 후 처리)
- 행 추가 금지 (MVP: fixed_row_count=true)

## 9. AI 분류기 한계 및 개선 포인트

### 9.1 현재 classifier.py 제한 사항

```python
# classifier.py:119 — 키워드 30개만 프롬프트에 주입 (제한)
kw_sample = dict(list(kw_dict.items())[:30])
```

**문제:** keyword_dict에 97개 키워드가 있지만, Claude 프롬프트에는 앞 30개만 주입됨.
CSV 순서대로 잘리므로 철거~목공 키워드만 들어가고 전기/설비/가구 키워드는 누락.

**개선 방향:**
- 옵션 A: 전체 97개 주입 (토큰 증가 허용 — 약 +500 토큰, 비용 미미)
- 옵션 B: 미매칭 품명과 관련된 키워드만 선택적 주입 (임베딩 유사도 기반)
- **MVP 권장: 옵션 A** (단순, 확실, 토큰 비용 무시 가능)

### 9.2 부분 매칭 함정

```python
# classifier.py:63-66 — 키워드가 품명에 포함되면 매칭
for keyword, mapping in kw_dict.items():
    if keyword in name:
        return mapping
```

**문제 케이스:**
- "강화마루" 키워드 → "강화유리" 품명에도 매칭 (강 포함)
- "유리" 키워드 → "욕실 유리문"에서 유리공사로 분류 (맞음)
- "도어" 키워드 → "도어락" 품명 → 창호공사로 분류 (맞음)
- "타일" 키워드 → "데코타일" → 타일공사 분류 (**틀림. 바닥공사여야**)

**개선 방향:** 키워드 매칭 시 word boundary 고려, 또는 우선순위 사전 구축.

### 9.3 신뢰도 보정 필요 케이스

| 상황 | 현재 동작 | 이상적 동작 |
|------|---------|-----------|
| 합계/소계 행 | Claude에게 전달 후 분류 시도 | keyword_dict에서 즉시 skip |
| 영문 약어 (EP, LVT, FCU) | 대부분 confidence 0.4~0.6 | 은어 사전 별도 등록 후 1.0 처리 |
| 수량=0인 행 | 분류 시도 | skip 권장 (빈 행) |

## 10. 컨펌 루프 UX 원칙

> ConfirmCard.tsx 설계 근거. 사용자가 5개 review_flag를 3분 안에 처리할 수 있어야 한다.

### 10.1 컨펌 카드 필수 표시 요소

```
┌─────────────────────────────────────────────┐
│ [검토 필요] 3/5                              │
│                                             │
│ 원본 품명: "EP도장 2차"                      │  ← item_name_raw (변경 불가)
│ 규격: 에폭시  |  단위: ㎡  |  수량: 20      │
│                                             │
│ AI 제안: 도장/도배공사 > 에폭시 도장         │  ← Claude 결과
│ 신뢰도: ██████░░░░ 65%                     │  ← confidence 시각화
│                                             │
│ [수락] [직접 입력] [모르겠음 → 나중에]       │
└─────────────────────────────────────────────┘
```

### 10.2 컨펌 버튼 3가지 원칙

| 버튼 | 동작 | DB 업데이트 |
|------|------|-----------|
| 수락 | AI 제안 그대로 확정 | confirmed_at 기록, review_flag 유지 |
| 직접 입력 | 대공종 드롭다운 + 세부공정 텍스트 | 사용자 값으로 덮어쓰기 |
| 나중에 | 다음 항목으로 넘어감 | 변경 없음 |

### 10.3 학습 루프 트리거 조건

사용자가 "직접 입력"으로 확정한 경우에만 keyword_dict에 신규 등록:

```
item_name_raw → (사용자 확정 공종) → keyword_dict INSERT
source = 'USER_CONFIRM'
confirm_count = 1
```

"수락"은 학습 안 함 — AI가 이미 아는 패턴이므로.

### 10.4 "모르겠음" 처리

현장에서도 모르는 항목이 존재한다. 강제 분류보다 `미분류` 상태 허용:
- 결과 엑셀에 `미분류`로 표기
- 나중에 전문가 검토 가능하도록 원본 유지
